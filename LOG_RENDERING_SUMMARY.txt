================================================================================
ACP CLIENT.LOG 内容未渲染排查 - 执行摘要
================================================================================

PROBLEM: "ACP Client.log 有很多内容没有渲染到 UI"

================================================================================
链路概览（4个阶段）
================================================================================

1. 日志产生
   ├─ Agent 进程 → acpClient.ts [onSessionUpdate]
   └─ 过滤: Tracer 只在 logLevel === Trace 时记录 ⚠️

2. 通知采集 & 解析
   ├─ acpClient.ts:250-282 [loadSession()]
   ├─ 完整通知列表保存在 notifications 数组
   └─ 12+ 种 sessionUpdate 类型（plan, tool_call, agent_message_chunk 等）

3. 历史记录序列化
   ├─ TurnBuilder.processNotification() [turnBuilder.ts:55-105]
   ├─ 仅处理 7 种类型 ✅: 
   │  - user_message_chunk, agent_message_chunk, agent_thought_chunk
   │  - tool_call, tool_call_update, plan
   └─ ❌ 5 种类型无处理:
      - available_commands_update, current_mode_update, config_option_update
      - session_info_update, usage_update

4. UI 流式渲染
   ├─ ChatParticipant.renderSessionUpdate() [acpChatParticipant.ts:599-802]
   ├─ 同样仅处理 7 种类型
   └─ 输出给 vscode.ChatResponseStream

================================================================================
五大确认缺口
================================================================================

缺口1: 日志级别门槛严格
   位置: tracer.ts:11
   原因: if(logLevel === LogLevel.Trace) 限制太严
   影响: 日志级别为 Info/Warn/Error 时，所有通知都不记录
   修复: 改为 logLevel !== LogLevel.Off

缺口2: 5种通知类型完全无处理
   位置: turnBuilder.ts:98-103, acpChatParticipant.ts:775-779
   原因: available_commands_update 等 5 种在 switch 中无 case
   影响: 这些通知产生但永远不会渲染或进入历史记录
   修复: 为各类型添加处理分支

缺口3: 工具输出字段提取不完整
   位置: chatRenderingUtils.ts:104-139
   原因: 仅提取 formatted_output/aggregated_output/output 中的第一个
   影响: 同时包含多字段时，其他字段被舍弃
   修复: 实现多字段合并策略

缺口4: PowerShell CLIXML 硬截断
   位置: chatRenderingUtils.ts:183-184
   原因: output.slice(0, markerIndex) 丢弃 CLIXML 标记之后的内容
   影响: Windows 工具长输出被截断
   修复: 改为解析 CLIXML 而非直接截断

缺口5: 会话恢复时的双重过滤
   位置: acpSessionManager.ts:400-426
   原因: Agent.loadSession() 返回的 notifications + TurnBuilder 过滤
   影响: 历史记录缺少通知
   修复: 验证 Agent 端过滤逻辑

================================================================================
最可能的缺口排序（按影响范围）
================================================================================

1️⃣ 缺口2 (5种通知无处理)
   - 影响最广: 这些通知永远看不到
   - 修复简单: 添加 case 分支

2️⃣ 缺口1 (日志级别)
   - 影响所有: 如果级别非 Trace，所有日志都丢
   - 修复简单: 改一行条件语句

3️⃣ 缺口4 (CLIXML 截断)
   - 影响 Windows: 长工具输出被截
   - 修复复杂: 需解析 CLIXML

4️⃣ 缺口3 (字段优先级)
   - 影响特殊场景: 仅在多字段情况
   - 修复中等: 需设计合并策略

5️⃣ 缺口5 (双重过滤)
   - 影响不确定: 取决于 Agent 实现
   - 修复困难: 需协调 Agent 端

================================================================================
诊断步骤（优先级）
================================================================================

[P1] 检查日志级别
     打开 VS Code 输出面板 "ACP Client" → 日志级别是否 Trace？

[P2] 对比通知计数
     acpClient.ts:253 加断点 → notifications.push()
     acpChatParticipant.ts:602 加断点 → tracer.trace()
     两处接收到的通知数是否相同？

[P3] 审查 5 种缺失类型
     是否项目实际产生了这 5 种通知？
     可在 acpClient.ts 的 loadSession 返回处输出日志

[P4] 测试工具输出
     使用包含长输出的工具，检查是否完整显示

================================================================================
关键文件职责明确
================================================================================

tracer.ts
  → 日志脱敏采集（严格日志级别控制）

acpClient.ts
  → Agent 连接 & 原始通知接收（无过滤）

acpSessionManager.ts
  → 会话生命周期管理 & 历史恢复

turnBuilder.ts
  → 通知序列化为历史记录（缺口: 5种无处理）

chatRenderingUtils.ts
  → 字段提取 & 格式化（缺口: 字段优先级、CLIXML截断）

acpChatParticipant.ts
  → UI 流式渲染执行（缺口: 5种无处理）
